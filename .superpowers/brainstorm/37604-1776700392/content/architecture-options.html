Architecture Approaches

Pick the right backend structure for the RAG upload pipeline

A

FastAPI + asyncio BackgroundTasks + PostgreSQL job tracking ★ Recommended

One FastAPI service handles uploads, runs ingestion as an async background task, and tracks job status in a documents table (status: pending → processing → done/failed). SSE endpoint watches the DB row and streams the completion event.

Pros

  • No extra services — only PostgreSQL needed
  • Dead simple to deploy (one container)
  • SSE fits naturally with asyncio

Cons

  • Job dies if server restarts mid-ingestion
  • Single worker — can't scale horizontally easily
Services: React → FastAPI → PostgreSQL + pgvector
B

FastAPI + Celery + Redis + PostgreSQL

Celery handles the background job queue with Redis as the broker. FastAPI enqueues the ingestion task after upload, and Celery workers process it reliably. SSE streams Celery task status back to the frontend.

Pros

  • Production-grade — jobs survive restarts
  • Horizontal scaling (add more workers)
  • Built-in retry logic

Cons

  • 3 extra services (Redis, Celery worker, Flower)
  • More complex docker-compose setup
  • Heavier for a first version
Services: React → FastAPI → Redis → Celery Worker → PostgreSQL + pgvector
C

FastAPI + PostgreSQL LISTEN/NOTIFY as job queue

Use PostgreSQL's native pub/sub (LISTEN/NOTIFY) to trigger ingestion on a separate worker process. No Redis needed — PostgreSQL does double duty as both job queue and vector store.

Pros

  • No Redis — PostgreSQL is the only external dependency
  • Jobs persist across restarts

Cons

  • More custom code (no battle-tested library)
  • PostgreSQL as task queue is less common — harder to debug
Services: React → FastAPI → PostgreSQL + pgvector ← Worker process
